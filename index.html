What are Devtools?

Browser dev tools (devtools for short) are a set of features integrated into modern web browsers, designed to aid web developers during the development process. While they are not building tools, they offer a number of crucial functions that aid the creation of apps, mainly:

  • Analysis of code and inner workings
  • Diagnosis of potential errors
  • Testing changes within the browser itself

Devtools are primarily used for working with HTML and CSS, but are also compatible with Javascript, Python and other similar languages.

It’s worth noting that IDEs are being slowly phased out for web development as browsers are progressively capable of doing more. A developer could potentially use nothing but the devtools provided by the browser to perform their duties. However, IDEs are still invaluable for learning the process of coding and possess quality of life features that browsers have yet to implement.

What follows is a brief overview of each devtool’s function. For more in-depth information about these devtools and their usage, please refer to the associated documentation accessible from a browser of your choice. You can also find links in the footer below.

Features

Inspect

The primary use of devtools is the ability to analyse a page or app’s source code. This is accomplished via the Inspect feature easily accessible via right-click menu. Inspecting allows a developer to view any given page’s HTML structure. The highlight tool enables selection of specific HTML elements, and the panel even allows for relocation of page elements via a drag-and-drop interface!

There is also a CSS panel which allows for the editing/addition/removal of style elements, with the added benefit of changing the state of the elements for the testing of pseudo-classes and other conditional styles. Lastly, the box model interface beneath the panel enables spacing and dimension modifications of individual elements.

Emulator

Next we have the emulator. As the name indicates, this devtool emulates several devices, allowing developers to preview their page on different screen sizes and ensure the responsiveness of their work, primarily on mobile-friendly pages.

It also features a throttler, which simulates different network conditions. This enables the testing of load speeds for content and other elements in a page. The media queries feature shows how they modify the page CSS.

Console

The console is the premiere diagnosis devtool, primarily when it comes to JavaScript. It will display any error messages stemming from issues in the code, as well as errors found when loading a page. The console is tied in with the network and JavaScript Debugger panels, as most reasons for loading errors can be found there (and accessible through clicking links).

Network

The network panel itself displays upload/download activity for page elements and allows for inspection of resource properties, such as size and content. It shows all the connections the browser parses while a page is loading, their speed and the order in which they are processed.

Each individual connection can be interacted with for further information, such as response time and encoding. You can save a log of your network activity via the preserve log function, and disable the cache to obtain more accurate load time simulations.

JavaScript Debugger

With the JavaScript debugger, you have the option to view JS errors on a line-by-line basis and even run additional code for the purposes of testing. It’s mainly accessed by clicking error messages found in the console. Additionally, it allows you to set breakpoints to analyse programmed variables with greater detail. Breakpoints are markers that "freeze" a program in a certain state while it’s being run, so specific behaviors can be checked without interfering with other functions.

Audits

The audits tool provides a way to diagnose issues with your page, whether those are performance or general experience related. The tool is powered by an external service called Lighthouse, which used to be exclusive to Google Chrome but now functions with other browsers. It’s a general use tool, which points out issues not necessarily visible from simple analysis of code. It could be considered a virtual second opinion/fresh pair of eyes.

Security

The security panel displays information pertaining to the security of a page’s connection. It can be used to display any certificates if present, and the connection protocol.

Storage

The storage tab displays reports on several types of application storage, namely:

  • Local storage :: Data stored locally and preserved between browser sessions.
  • Session storage :: Identical to local storage, but the data is lost when the current session ends.
  • Cache storage :: Data which persists until specifically deleted.
  • Cookies :: Cookies created by the current page or any of its iframes. Network-generated cookies are also displayed, but only while the report is open.
  • IndexedDB :: Data generated by any IndexedDB present within the page or its iframes. This is used for storing mass amounts of data, generally files that aren’t stored externally.

Differences across browsers

In the past, certain browsers, namely Internet Explorer, had proprietary HTML tags which made pages not render properly when viewed through other browsers’ tools. As time went on, the paradigm shifted to a point where it no longer matters which browser is used.

For example, Firefox offers more artistic/design oriented tools, whereas Chrome keeps an air of practicality. For Apple-oriented developers, Safari is the obvious choice.

Beyond nomenclature for specific features and the clear differences in UI design, it’s completely up to preference.
